# Role permissions

Veridian Motion uses **four staff roles** plus separate **patient** kiosk
access. Permissions are enforced through the application and **row-level
security (RLS)** scoped by **facility membership**. Staff with access to
multiple facilities can **switch between facilities** in the dashboard.

<Note>
  Staff require **email + password** and **TOTP MFA** (enforced per deployment).
  Roles are granted only by an **admin**.
</Note>

## Role summaries

### Physical Therapist (`pt`)

Primary **clinical user** for a **facility**. Reviews **assigned**
assessments only — red flags, intake, AI exercise plan. Edits, **approves**,
notes, messages patients, exports PDFs. Clinical data scoped to **their
facility** via RLS.

### Administrator (`admin`)

**Operational and clinical administration** for a facility. Enrolls patients,
assigns cases, resets PINs, grants roles, MFA reset, roster, audit logs,
**Communication Record** access, and (when enabled) **admin analytics**.

### Warden / facility oversight (`warden`)

**Operational visibility only** — enrollment counts, confirmation that care
workflows are active, and **safety escalations**. **No clinical PHI**: no
assessments, AI output, exercise plans, clinical notes, message bodies, or
patient-identifiable roster views. HIPAA **minimum-necessary** by design.

### IT Admin (`it_admin`)

**Technical administration** — security settings, compliance tooling, MFA
break-glass. No clinical review or patient enrollment by default.

## Patient (kiosk — not a staff role)

| Action | Allowed |
| --- | --- |
| Sign in with Patient ID + PIN | Yes |
| Complete patient portal intake | Yes |
| View own submission status | Yes |
| View approved plan after PT approval | Yes |
| View PT notes marked **shared with patient** | Yes |
| Reply in secure messaging threads | Yes |
| View AI output before PT approval | **No** |
| Self sign-up or change PIN | **No** |
| Access other patients' data | **No** |

## Multi-facility isolation

- Each facility's **patients and staff data** are isolated through **RLS** by
  **facility membership**
- **`pt`** clinicians see clinical data for **their facility only** (per assignment)
- Cross-facility patient access is denied at the database layer

## Enforcement

1. **Application UI and API** — role checks on staff actions
2. **Row-level security** — facility-scoped queries; cross-patient and cross-facility access denied even if UI is bypassed
3. **Hash-chained audit log** — sensitive actions recorded without PHI in payload; **Communication Record** access is audit-logged
